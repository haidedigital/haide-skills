# QA & SEO Risk Management Skill

## Philosophy

**Paranoia is a virtue. Assume it will break. Build systems to catch it when it does.**

Premium QA isn't about finding bugs‚Äîit's about preventing catastrophic SEO failures that could cost clients millions. At $25k+/month, clients expect bulletproof processes, pre-launch safety checks, and instant crisis response. One bad launch can destroy a year of organic growth work.

---

## When to Use This Skill

**Primary scenarios:**
- Before any site launch, redesign, or major change
- Planning site migrations (domain, platform, URL structure)
- Reviewing content production at scale (AI-generated content)
- Setting up automated monitoring and alerts
- After algorithm updates (assess impact and response)
- When rankings or traffic drop unexpectedly
- Before launching new features or functionality
- Implementing large-scale technical changes
- Auditing third-party vendor work
- Establishing quality gates for premium delivery

**Premium context triggers:**
- Client planning $10M+ site migration
- Executive nervous about redesign risk ("will this hurt our traffic?")
- Sudden traffic drop (crisis mode)
- Client asks "what's our rollback plan?"
- High-stakes launch (new product, rebrand, etc.)
- Protecting organic channel worth $5M+ annual value

**Integration with other skills:**
- Strategic & Technical SEO: Technical foundation for QA protocols
- Project Management: Quality gates in delivery process
- Automation & Systems: Automated monitoring and alerts
- Data & Measurement: Performance tracking, anomaly detection
- Client Communication: Crisis communication, risk reporting
- Commercial Judgment: Risk vs reward analysis, investment protection

## Core Competencies

### 1. Pre-Launch Safety Protocols

**Premium Standard**: Nothing goes live without passing comprehensive pre-launch checklist.

**The Pre-Launch Checklist** (Master):

```
TECHNICAL SEO VALIDATION:
‚òê Robots.txt correct (not blocking important pages)
‚òê XML sitemaps generated and submitted
‚òê Canonical tags correct (self-referencing or intentional)
‚òê No unintentional noindex/nofollow tags
‚òê Hreflang implemented correctly (if international)
‚òê Redirects mapped and tested (301s, not 302s)
‚òê No redirect chains or loops
‚òê Internal links functional (no 404s)
‚òê Site speed acceptable (Core Web Vitals passing)
‚òê Mobile-friendly (responsive design working)
‚òê HTTPS implemented correctly (no mixed content)
‚òê Structured data (schema markup) validated
‚òê Pagination implemented correctly (if applicable)
‚òê Faceted navigation not creating duplicate content
‚òê URL parameters handled correctly (Google Search Console)

CONTENT VALIDATION:
‚òê All pages have unique title tags
‚òê All pages have unique meta descriptions
‚òê H1 tags present and descriptive
‚òê Images have alt text
‚òê Content is indexed (no accidental noindex)
‚òê No thin content pages (<300 words without good reason)
‚òê No duplicate content issues
‚òê Canonical URLs correct
‚òê Internal linking structure preserved or improved

ON-PAGE SEO:
‚òê Target keywords in titles, headings, content
‚òê URL structure SEO-friendly
‚òê Breadcrumbs implemented
‚òê Related content recommendations working
‚òê CTAs and conversion elements present

INDEXING & CRAWLING:
‚òê Important pages are indexable
‚òê Unimportant pages are noindexed (admin, filters, etc.)
‚òê Crawl budget optimized (no wasted crawling)
‚òê Log file analysis completed (what is Googlebot crawling?)

USER EXPERIENCE:
‚òê Page load time <3 seconds (ideally <2 seconds)
‚òê Largest Contentful Paint (LCP) <2.5 seconds
‚òê First Input Delay (FID) <100ms
‚òê Cumulative Layout Shift (CLS) <0.1
‚òê No intrusive interstitials
‚òê Readable font sizes (mobile)
‚òê Tap targets appropriately sized (mobile)

CROSS-BROWSER & DEVICE TESTING:
‚òê Works in Chrome, Firefox, Safari, Edge
‚òê Works on iOS and Android
‚òê Works on desktop, tablet, mobile
‚òê Responsive design functioning correctly

TRACKING & ANALYTICS:
‚òê GA4 tracking code installed and firing
‚òê Google Search Console verified
‚òê Conversion tracking working
‚òê Event tracking configured
‚òê Goals/conversions marked correctly

EXTERNAL INTEGRATIONS:
‚òê Third-party scripts not blocking rendering
‚òê CDN functioning correctly
‚òê Email forms submitting correctly
‚òê Payment processing working (if applicable)
‚òê Chat widgets working
‚òê Social sharing buttons functional

STAGING ENVIRONMENT VALIDATION:
‚òê Tested on staging environment first
‚òê Staging environment hidden from search engines (noindex)
‚òê All issues fixed on staging before production deploy

ROLLBACK PLAN:
‚òê Backup of current site taken
‚òê Rollback procedure documented and tested
‚òê Database backup (if applicable)
‚òê DNS changes reversible
‚òê Can revert within 1 hour if needed
```

**Premium Pre-Launch Process**:

```
T-7 days: Staging environment ready
- Complete QA checklist on staging
- Fix all critical issues
- Stakeholder review and approval

T-3 days: Pre-launch technical audit
- External SEO audit (fresh eyes)
- Performance testing (load testing if high-traffic site)
- Final stakeholder sign-off

T-1 day: Final checks
- Re-run QA checklist
- Verify backup and rollback plan
- Communication plan ready (internal team + client)

Launch day:
- Deploy during low-traffic period (e.g., Sunday 2am)
- Monitor in real-time (first 2-4 hours critical)
- Check: Indexing, crawling, traffic, rankings, conversions

T+1 day: Post-launch validation
- GSC: Check for new errors
- Analytics: Traffic normal?
- Rankings: Spot-check key pages
- User testing: Everything functional?

T+7 days: Performance review
- Compare to pre-launch baselines
- Address any issues discovered
- Document lessons learned
```

### 2. Site Migration Risk Management

**Premium Standard**: Protect client's organic investment during high-risk migrations.

**Types of Migrations** (by risk level):

```
LOW RISK:
- Content updates (refreshing existing pages)
- Adding new pages (not changing existing)
- Minor design tweaks (CSS changes only)

MEDIUM RISK:
- Template changes (affecting many pages)
- URL structure changes (with proper redirects)
- CMS migration (same URLs)

HIGH RISK:
- Domain migration (new domain)
- Platform migration (e.g., Shopify to custom)
- Large-scale URL changes
- Site redesign + migration combined

EXTREME RISK:
- Domain + platform + URL structure all changing
- International expansion (new ccTLDs, hreflang)
- Site consolidation (merging multiple sites)
- Any migration worth $5M+ in annual organic value
```

**Migration Safety Framework**:

**Phase 1: Pre-Migration Audit** (4-8 weeks before)

```
Document current state:
‚òê Export all URLs (current site structure)
‚òê Document current rankings (top 500-1000 keywords)
‚òê Export baseline analytics (traffic, conversions, revenue)
‚òê Identify high-value pages (revenue drivers, high traffic)
‚òê Identify low-value pages (candidates for pruning)
‚òê Document all backlinks (Ahrefs, Semrush)
‚òê Export internal link structure
‚òê Crawl site (Screaming Frog) - baseline data

Assess risk:
‚òê What's changing? (domain, URLs, platform, content, design?)
‚òê What's the value at risk? (annual organic revenue)
‚òê What's the downside scenario? (worst case traffic drop estimate)
‚òê Is migration necessary? (what's the business case?)
‚òê Can migration be staged? (reduce risk by doing in phases)
```

**Phase 2: Migration Planning** (3-6 weeks before)

```
URL mapping:
‚òê Map old URLs to new URLs (1:1 mapping)
‚òê Identify URLs to redirect (301 redirects)
‚òê Identify URLs to consolidate (multiple old ‚Üí one new)
‚òê Identify URLs to prune (delete/noindex)
‚òê Validate mapping (no broken links)

Technical planning:
‚òê Redirect implementation plan (server-side, not JavaScript)
‚òê XML sitemap for new site
‚òê Robots.txt for new site
‚òê Canonical tag strategy
‚òê Internal linking preservation/improvement

Content planning:
‚òê Content that needs updating (refresh for new site)
‚òê Title tags and meta descriptions (optimize for new URLs)
‚òê Schema markup (re-implement or improve)

Stakeholder alignment:
‚òê Client understands risks (potential temporary drop)
‚òê Timeline agreed (launch date, monitoring period)
‚òê Success criteria defined (what does success look like?)
‚òê Communication plan (who needs to know what, when?)
```

**Phase 3: Staging & Testing** (2-4 weeks before)

```
Build and test on staging:
‚òê New site built on staging environment
‚òê All content migrated
‚òê Redirects implemented and tested
‚òê Internal links functional
‚òê Forms and functionality working
‚òê Analytics tracking configured
‚òê Run full QA checklist (from Pre-Launch section above)

Validate redirects:
‚òê Test random sample of redirects (100+ URLs)
‚òê Check for redirect chains (old ‚Üí interim ‚Üí new = bad)
‚òê Check for redirect loops (old ‚Üí new ‚Üí old = very bad)
‚òê Verify 301 status codes (not 302)

Performance testing:
‚òê Page speed acceptable (ideally better than current)
‚òê Core Web Vitals passing
‚òê Mobile-friendly

External validation:
‚òê SEO expert review (external audit)
‚òê Developer QA (functionality testing)
‚òê User acceptance testing (real users test staging)
```

**Phase 4: Migration Execution** (Launch day)

```
Pre-launch:
‚òê Backup current site (full backup)
‚òê Backup database (if applicable)
‚òê Document current DNS settings
‚òê Communicate with team (everyone on standby)

Launch:
‚òê Deploy new site (during low-traffic period)
‚òê Implement redirects (server-level)
‚òê Update DNS (if domain change)
‚òê Submit new XML sitemap to GSC
‚òê Update robots.txt (if needed)
‚òê Monitor in real-time (first 2-4 hours critical)

Immediate checks:
‚òê Spot-check 20-50 URLs (do they work?)
‚òê Test redirects (do old URLs redirect correctly?)
‚òê Check GSC (any immediate errors?)
‚òê Check analytics (traffic flowing?)
‚òê Check conversion tracking (still working?)
```

**Phase 5: Post-Migration Monitoring** (4-12 weeks after)

```
Daily monitoring (first week):
‚òê GSC errors (any new 404s, 5xx errors?)
‚òê Traffic levels (compared to pre-migration baseline)
‚òê Rankings (spot-check key pages)
‚òê Conversions (maintaining pre-migration levels?)

Weekly monitoring (weeks 2-4):
‚òê Indexing status (are new URLs indexed?)
‚òê Ranking recovery (returning to pre-migration levels?)
‚òê Traffic trends (stabilizing or still volatile?)
‚òê Backlink discovery (are backlinks resolving to new URLs?)

Monthly monitoring (weeks 5-12):
‚òê Full ranking comparison (vs pre-migration baseline)
‚òê Traffic and revenue comparison
‚òê Identify pages that didn't recover (investigate and fix)
‚òê Client reporting (migration success metrics)

Success criteria:
‚òê 90%+ of pre-migration traffic maintained after 90 days
‚òê 90%+ of pre-migration rankings maintained
‚òê 95%+ of pre-migration conversions maintained
‚òê No significant technical errors in GSC
```

**Migration Crisis Response Plan**:

```
If traffic drops >20% in first week:

IMMEDIATE (within 24 hours):
1. Check GSC for errors (404s, 5xx, indexing issues)
2. Verify redirects working correctly (random sample test)
3. Check analytics tracking (is tracking broken?)
4. Review log files (is Googlebot crawling?)
5. Spot-check high-value pages (are they indexed and ranking?)

IF PROBLEM IDENTIFIED:
- Fix immediately (redirect issue, indexing problem, etc.)
- Communicate to client (transparency)
- Document fix and monitor recovery

IF NO PROBLEM IDENTIFIED:
- May be temporary Google volatility (migration signal)
- Continue monitoring closely
- If no recovery after 2 weeks, investigate deeper

ROLLBACK DECISION:
If traffic drops >50% and cause unknown:
- Consider rollback to old site
- Requires stakeholder decision (rollback = delay)
- Only if critical business impact
```

### 3. Keyword Cannibalization Detection & Resolution

**Premium Standard**: Proactively prevent and fix pages competing against each other.

**What is Cannibalization?**

When multiple pages on your site target the same keyword, they compete against each other. Google doesn't know which to rank, so neither ranks well.

**Cannibalization Detection Process**:

**Step 1: Identify potential cannibalization**

```
Method 1: GSC Performance Report
1. Go to GSC ‚Üí Performance ‚Üí Search Results
2. Filter by query (high-value keyword)
3. Group by: Page
4. Look for: Multiple pages ranking for same query

Example:
Query: "project management software"
- /products/pm-software: 47 clicks
- /blog/best-pm-software: 23 clicks
- /resources/pm-software-guide: 12 clicks

Result: 3 pages cannibalizing each other (total 82 clicks split)

Method 2: Rank Tracking Tools
1. Use Ahrefs or Semrush Position Tracking
2. Filter by keyword
3. Look for: Multiple URLs ranking for same keyword
4. Check: Are rankings volatile? (sign of cannibalization)

Method 3: Site Search
1. Google: site:yourdomain.com "target keyword"
2. See which pages rank
3. If multiple similar pages ‚Üí potential cannibalization
```

**Step 2: Assess severity**

```
HIGH PRIORITY (fix immediately):
- Both pages important (high traffic or conversion)
- Keyword is high-value (commercial intent, high volume)
- Rankings are declining (losing to competitors)

MEDIUM PRIORITY (fix soon):
- One page important, one less important
- Keyword is medium-value
- Rankings stable but could be better

LOW PRIORITY (monitor):
- Both pages low-value
- Keyword is low-volume
- No business impact
```

**Step 3: Choose resolution strategy**

```
OPTION 1: CONSOLIDATE (merge into one strong page)
Best when: Both pages have similar content

Process:
1. Choose primary page (usually the better-performing one)
2. Migrate best content from secondary page to primary page
3. 301 redirect secondary page ‚Üí primary page
4. Update internal links to point to primary page

Example:
- Primary: /products/pm-software (product page, high conversion)
- Secondary: /blog/best-pm-software (informational)
- Action: Add comparison content from blog to product page, redirect blog ‚Üí product page

OPTION 2: DIFFERENTIATE (make pages target different keywords)
Best when: Pages serve different intents

Process:
1. Analyze search intent for the keyword
2. Assign primary keyword to best-matching page
3. Optimize other page for different keyword (long-tail variant)
4. Update title, headings, content to differentiate

Example:
- Page 1: /products/pm-software ‚Üí Target "project management software" (commercial intent)
- Page 2: /blog/what-is-pm-software ‚Üí Target "what is project management software" (informational intent)
- Action: Differentiate intent, optimize each for different keyword

OPTION 3: DELETE/NOINDEX (remove weaker page)
Best when: One page is clearly superior, other adds no value

Process:
1. Identify weaker page (low traffic, low conversion, thin content)
2. Delete page OR noindex page
3. 301 redirect ‚Üí stronger page (if has backlinks)
4. Update internal links

OPTION 4: CANONICALIZE (declare primary version)
Best when: Pages are intentionally similar (e.g., product variations, paginated content)

Process:
1. Identify primary version
2. Add canonical tag to secondary pages ‚Üí primary page
3. Signals to Google: "These pages are similar, rank the canonical"
```

**Cannibalization Prevention**:

```
BEFORE creating new content:

1. Check if existing page already targets keyword
   - Search: site:yourdomain.com "target keyword"
   - Check GSC: Does any page already rank for this?

2. If existing page exists:
   - Update/refresh existing page (don't create duplicate)
   - OR intentionally differentiate (target different intent/keyword)

3. If creating new page:
   - Choose unique target keyword
   - Differentiate from existing content
   - Consider: Could this be a section on existing page instead?

CONTENT BRIEF CHECKLIST:
‚òê Target keyword is unique (not already targeted by another page)
‚òê Search intent is clear (informational, commercial, transactional)
‚òê Differentiation from similar content defined
‚òê Internal linking strategy planned (how this fits into site structure)
```

### 4. Regression Testing & Change Management

**Premium Standard**: Ensure fixing this doesn't break that.

**What is Regression Testing?**

Testing that a change (update, fix, new feature) doesn't break existing functionality.

**Regression Testing Protocols**:

**Scenario 1: Template change** (affects multiple pages)

```
Change: Updating blog post template

Regression test:
‚òê Crawl sample of blog posts (10-20 URLs)
‚òê Verify: Title tags still correct
‚òê Verify: Meta descriptions still correct
‚òê Verify: H1 tags present
‚òê Verify: Schema markup still present
‚òê Verify: Internal links functional
‚òê Verify: Images still display
‚òê Verify: Page speed not degraded
‚òê Check GSC: No new errors after deploy
‚òê Spot-check rankings (do they drop?)
```

**Scenario 2: Redirect implementation**

```
Change: Adding 301 redirects for old URLs

Regression test:
‚òê Test random sample of redirects (50+ URLs)
‚òê Verify: 301 status code (not 302 or other)
‚òê Verify: No redirect chains (A ‚Üí B ‚Üí C)
‚òê Verify: No redirect loops (A ‚Üí B ‚Üí A)
‚òê Verify: Target pages are indexed and functional
‚òê Check GSC: Monitor 404 errors (should decrease)
‚òê Monitor traffic: Pages receiving redirected traffic?
```

**Scenario 3: JavaScript changes** (potentially affects SEO)

```
Change: Adding new JavaScript functionality

Regression test:
‚òê Test with JavaScript disabled (is content still visible?)
‚òê Check page source (is critical content in HTML?)
‚òê Use "Fetch as Google" in GSC (does Google see content?)
‚òê Check rendering (GSC ‚Üí URL Inspection ‚Üí View crawled page)
‚òê Verify: Internal links crawlable (not JavaScript-only)
‚òê Verify: Page speed not significantly degraded
‚òê Check mobile: Functionality works on mobile
```

**Scenario 4: CMS plugin/module update**

```
Change: Updating Yoast SEO plugin (or similar)

Regression test:
‚òê Test on staging first (never update plugins in production without testing)
‚òê Verify: Title tags still generating correctly
‚òê Verify: Meta descriptions still generating
‚òê Verify: Schema markup still present
‚òê Verify: XML sitemap still generating
‚òê Verify: Robots.txt not changed
‚òê Verify: Canonical tags still correct
‚òê Crawl site: Check for any new errors
```

**Change Management Process** (for premium clients):

```
CHANGE REQUEST:
- Document: What's changing and why?
- Assess: SEO impact (low, medium, high risk?)
- Approve: Stakeholder sign-off (client + internal)

TESTING:
- Test on staging environment
- Run regression test checklist
- Fix any issues discovered
- External review (if high-risk change)

DEPLOYMENT:
- Deploy during low-traffic period
- Monitor in real-time (first hour critical)
- Have rollback plan ready

POST-DEPLOYMENT:
- Verify change successful
- Monitor for 24-48 hours (check analytics, GSC)
- Document any issues and resolutions
- Report to stakeholders (change summary)
```

### 5. Crisis Management & Rapid Response

**Premium Standard**: When traffic drops 30%+, respond within 1 hour.

**Traffic Drop Crisis Protocol**:

**Phase 1: Immediate Triage** (within 1 hour)

```
Step 1: Confirm the drop is real
‚òê Check analytics: Is tracking working? (real-time shows traffic?)
‚òê Compare: Same period last week, last month, last year
‚òê Check: All channels down or just organic?
‚òê Verify: Conversion tracking still working?

Step 2: Check for external factors
‚òê Seasonality: Is this expected? (holidays, industry events)
‚òê Google algorithm update: Check Moz, Search Engine Journal, Barry Schwartz
‚òê Competitor changes: Did competitor suddenly rank better?
‚òê News event: Is there breaking news affecting search behavior?

Step 3: Check for technical issues
‚òê GSC: Any new errors? (Coverage issues, Mobile usability, Core Web Vitals)
‚òê Site status: Is site up? (server errors?)
‚òê Robots.txt: Was it accidentally changed?
‚òê Noindex tags: Were pages accidentally noindexed?
‚òê Redirects: Were redirects broken?
‚òê DNS: Any DNS issues?

Step 4: Check for manual actions or penalties
‚òê GSC ‚Üí Security & Manual Actions: Any manual actions?
‚òê Suspicious link profile: Negative SEO attack? (check Ahrefs for spam links)
‚òê Over-optimization: Did we trigger algorithmic filter?
```

**Phase 2: Diagnosis** (hours 2-4)

```
Determine scope:
‚òê All pages down or specific pages?
‚òê All keywords down or specific keywords?
‚òê All search engines or just Google?

Analyze patterns:
‚òê Which pages lost traffic? (export from analytics)
‚òê Which keywords lost rankings? (check rank tracker)
‚òê What do affected pages have in common? (template, topic, optimization?)

Review recent changes:
‚òê What changed in last 7-14 days? (site updates, content, redirects)
‚òê Any deployments or updates?
‚òê Any third-party changes? (CDN, hosting, CMS updates)

Hypothesis formation:
Based on data, form hypothesis about cause:
- Algorithm update targeting [specific issue]
- Technical issue with [specific area]
- Penalty for [specific violation]
- Competitor gain (we didn't drop, they rose)
```

**Phase 3: Response Plan** (hours 4-24)

```
If technical issue identified:
‚Üí Fix immediately
‚Üí Test fix on staging if possible
‚Üí Deploy fix to production
‚Üí Monitor for recovery (2-7 days typically)

If algorithm update:
‚Üí Analyze: What did algorithm target? (quality, links, user experience?)
‚Üí Plan: How to address algorithm's target?
‚Üí Implement: Fixes aligned with algorithm intent
‚Üí Monitor: Recovery may take 4-12 weeks

If manual action or penalty:
‚Üí Review: What triggered the manual action?
‚Üí Fix: Remove/disavow problematic links or fix issue
‚Üí Request reconsideration: Submit reconsideration request in GSC
‚Üí Monitor: Recovery after Google reviews (2-4 weeks)

If competitor gain:
‚Üí Analyze: What did competitor do better?
‚Üí Plan: How to counter competitive advantage?
‚Üí Implement: Content improvements, link acquisition, etc.
‚Üí Monitor: Competitive positioning (ongoing)
```

**Phase 4: Communication** (throughout)

```
Internal communication:
- Hour 1: Alert team to drop
- Hour 4: Share diagnosis and response plan
- Daily: Status updates until resolved

Client communication (premium clients expect transparency):
- Hour 2-4: Initial notification ("We're aware of traffic drop, investigating")
- Day 1: Diagnosis and response plan ("Here's what we found, here's our plan")
- Weekly: Progress updates until recovery
- Post-recovery: Summary and lessons learned

Communication template:

Subject: Traffic Drop Alert - [Client Name] - [Date]

Hi [Client],

We've detected a [X%] decrease in organic traffic starting [date/time].

Current status:
- Confirmed: Traffic drop is real (not tracking issue)
- Scope: [All pages / Specific pages / Specific keywords]
- Cause: [Hypothesis based on investigation]

Our response:
1. [Immediate action taken]
2. [Planned fix with timeline]
3. [Expected recovery timeline]

We're monitoring closely and will update you [daily/every other day] until resolved.

[Your signature]
```

**Phase 5: Recovery Validation** (weeks-months)

```
Monitor recovery:
‚òê Daily: Traffic levels, key keyword rankings
‚òê Weekly: Comparison to pre-drop baseline
‚òê Monthly: Full performance review

Validate recovery:
‚òê Traffic returned to 90%+ of pre-drop levels
‚òê Rankings recovered for priority keywords
‚òê Conversions maintained
‚òê No lingering technical issues

Document lessons learned:
‚òê What caused the drop?
‚òê How was it detected?
‚òê How was it fixed?
‚òê How can we prevent it in future?
‚òê Update protocols based on learnings
```

### 6. Automated Monitoring & Alert Systems

**Premium Standard**: Catch issues before they become crises.

**Monitoring Tools Setup**:

```
CRITICAL MONITORS (check 24/7, alert immediately):

1. Uptime monitoring
   - Tool: UptimeRobot, Pingdom, StatusCake
   - Check: Every 1-5 minutes
   - Alert: If site down for >2 minutes
   - Notify: Team via SMS/Slack immediately

2. GSC error monitoring
   - Tool: Google Search Console
   - Check: Daily automated check
   - Alert: If >10 new 404 errors or Coverage errors
   - Notify: Email daily summary, Slack if critical

3. Robots.txt monitoring
   - Tool: ContentKing, Little Warden
   - Check: Every hour
   - Alert: If robots.txt changes (could accidentally block Google)
   - Notify: Immediate Slack alert

4. Ranking monitoring (high-value keywords)
   - Tool: Ahrefs, Semrush, AccuRanker
   - Check: Daily
   - Alert: If top 10 keywords drop out of top 10
   - Notify: Email daily summary

5. Traffic anomaly detection
   - Tool: Google Analytics 4 (custom alert)
   - Check: Hourly
   - Alert: If traffic drops >20% compared to previous week same day/hour
   - Notify: Email + Slack

IMPORTANT MONITORS (check daily, alert if issue):

6. Index coverage monitoring
   - Tool: GSC or Screaming Frog (scheduled crawl)
   - Check: Daily
   - Alert: If >5% of important pages deindexed
   - Notify: Email daily

7. Core Web Vitals monitoring
   - Tool: GSC or CrUX Dashboard
   - Check: Weekly
   - Alert: If passing rate drops below 75%
```

```
‚òê Robots.txt correct (not blocking important pages)

‚òê All pages have unique title tags
‚òê All pages have unique meta descriptions
‚òê H1 tags present and descriptive
‚òê Images have alt text
‚òê Schema markup present and valid

Technical:
‚òê Page speed acceptable (<3 seconds load time)
‚òê Mobile-friendly (responsive design)
‚òê Core Web Vitals passing
‚òê No JavaScript rendering issues

‚òê Important pages well-linked
```

## Common Scenarios & Solutions

### Scenario 1: "We're migrating our site next week. What could go wrong?"

**Answer: Comprehensive risk assessment**:

```
HIGH-RISK AREAS:

1. REDIRECTS
   Risk: Broken or missing redirects ‚Üí 404 errors, traffic loss
   Mitigation:
   - Map every old URL to new URL (1:1 mapping)
   - Test random sample of 100+ redirects
   - Check for redirect chains and loops
   - Verify 301 (not 302) status codes

2. INDEXING
   Risk: New pages not indexed, old pages still indexed
   Mitigation:
   - Submit new XML sitemap immediately after launch
   - Request re-indexing for high-value pages (GSC)
   - Monitor index coverage daily

3. CONTENT LOSS
   Risk: Content doesn't migrate correctly
   Mitigation:
   - Content audit before migration (what exists now?)
   - Verify all content migrated (compare pre and post)
   - Check for missing images, broken formatting

4. TECHNICAL ISSUES
   Risk: Site speed, mobile issues, broken functionality
   Mitigation:
   - Test site speed on staging (should be same or better)
   - Test mobile version thoroughly
   - Run full pre-launch checklist

5. ANALYTICS TRACKING
   Risk: Tracking breaks, can't measure impact
   Mitigation:
   - Reinstall GA4 tracking on new platform
   - Test tracking before launch (submit test conversion)
   - Verify tracking immediately after launch

ROLLBACK PLAN:
- Full backup of old site
- DNS can be reverted within 1 hour
- Redirects can be reversed
- If traffic drops >50% in first week with unknown cause, consider rollback

MONITORING PLAN:
- Real-time monitoring first 4 hours post-launch
- Daily checks first week
- Weekly checks next 3 weeks
- Full comparison after 90 days
```

### Scenario 3: "How do we prevent keyword cannibalization?"

**Answer: Content planning process**:

```
BEFORE creating new content:

Step 1: Check if topic already covered
- Search: site:yourdomain.com "target keyword"
- Check GSC: Performance ‚Üí Queries ‚Üí search for keyword
- Review content inventory

Step 2: If existing page found:
Option A: Update existing page (don't create duplicate)
- Refresh with new information
- Expand with more depth
- Optimize for SEO

Option B: Create new page IF intentionally different
- Different search intent (informational vs commercial)
- Different audience (beginner vs expert)
- Different format (blog post vs guide vs product page)
- Different keyword (long-tail variation)

Step 3: If creating new page:
- Document: Target keyword is [X], existing page targets [Y]
- Differentiate: This page will focus on [different angle]
- Internal linking: Link between related pages appropriately

ONGOING MONITORING:
- Monthly cannibalization audit (check GSC for multiple pages ranking for same query)
- Fix proactively before it hurts rankings
```

```
AUTOMATED QA (tools):
‚òê Plagiarism check: Copyscape, Grammarly
```

## Key Principles

1. **Paranoia is a virtue**
   - Assume it will break
   - Build systems to catch it when it does
   - Better safe than sorry

2. **Prevention > Detection > Response**
   - Best: Prevent problems (pre-launch QA)
   - Good: Detect problems fast (automated monitoring)
   - Acceptable: Respond quickly (crisis protocol)

3. **Always have an undo button**
   - Rollback plan for every deployment
   - Backups before major changes
   - Can revert within 1 hour

4. **Test everything, assume nothing**
   - Staging environment for all changes
   - Multiple browsers and devices
   - Comprehensive checklists

5. **Multiple layers of review**
   - Self-check ‚Üí Peer ‚Üí QA ‚Üí Expert
   - Appropriate rigor for risk level
   - Fresh eyes catch what you miss

6. **Automate monitoring, not quality**
   - Automate: Uptime checks, error alerts, traffic monitoring
   - Don't automate: Content review, strategic decisions
   - Humans + automation = best results

7. **Communicate transparently**
   - Tell client about risks before launch
   - Alert client to issues immediately
   - Honesty builds trust

8. **Learn from every incident**
   - Post-mortem after every failure
   - Document lessons learned
   - Update processes to prevent recurrence

9. **Quality is everyone's job**
   - Not just QA team's responsibility
   - Every team member does self-check
   - Culture of quality and excellence

10. **Measure quality, improve continuously**
    - Track: Issues found per sprint, time to resolution, incident rate
    - Set targets: Zero critical bugs deployed to production
    - Iterate: Make QA process better every month
